8//
9// Data structures for storing variable assignment information in LLVM. In the
10// dbg.value design, a dbg.value intrinsic specifies the position in a block
11// a source variable take on an LLVM Value:
12//
13// %foo = add i32 1, %0
14// dbg.value(metadata i32 %foo, ...)
15// %bar = void call @ext(%foo);
16//
17// and all information is stored in the Value / Metadata hierarchy defined
18// elsewhere in LLVM. In the "DbgRecord" design, each instruction /may/ have a
19// connection with a DbgMarker, which identifies a position immediately before
20// the instruction, and each DbgMarker /may/ then have connections to DbgRecords
21// which record the variable assignment information. To illustrate:
22//
23// %foo = add i32 1, %0
24// ; foo->DebugMarker == nullptr
25// ;; There are no variable assignments / debug records "in front" of
26// ;; the instruction for %foo, therefore it has no DebugMarker.
27// %bar = void call @ext(%foo)
28// ; bar->DebugMarker = {
29// ; StoredDbgRecords = {
30// ; DbgVariableRecord(metadata i32 %foo, ...)
31// ; }
32// ; }
33// ;; There is a debug-info record in front of the %bar instruction,
34// ;; thus it points at a DbgMarker object. That DbgMarker contains a
35// ;; DbgVariableRecord in its ilist, storing the equivalent information
36// ;; to the dbg.value above: the Value, DILocalVariable, etc.
37//
38// This structure separates the two concerns of the position of the debug-info
39// in the function, and the Value that it refers to. It also creates a new
40// "place" in-between the Value / Metadata hierarchy where we can customise
41// storage and allocation techniques to better suite debug-info workloads.
42// NB: as of the initial prototype, none of that has actually been attempted
43// yet.
44//
45//===----------------------------------------------------------------------===//

87 /// Construct from an \a MDNode.
88 ///

92 /// IR.

582/// Per-instruction record of debug-info. If an Instruction is the position of
583/// some debugging information, it points at a DbgMarker storing that info. Each
584/// marker points back at the instruction that owns it. Various utilities are
585/// provided for manipulating the DbgRecords contained within this marker.
586///
587/// This class has a rough surface area, because it's needed to preserve the
588/// one arefact that we can't yet eliminate from the intrinsic / dbg.value
589/// debug-info design: the order of records is significant, and duplicates can
590/// exist. Thus, if one has a run of debug-info records such as:
591/// dbg.value(...
592/// %foo = barinst
593/// dbg.value(...
594/// and remove barinst, then the dbg.values must be preserved in the correct
595/// order. Hence, the use of iterators to select positions to insert things
596/// into, or the occasional InsertAtHead parameter indicating that new records
597/// should go at the start of the list.
598///
599/// There are only five or six places in LLVM that truly rely on this ordering,
600/// which we can improve in the future. Additionally, many improvements in the
601/// way that debug-info is stored can be achieved in this class, at a future
602/// date.
603class DbgMarker {
604public:
605 DbgMarker() = default;
606 /// Link back to the Instruction that owns this marker. Can be null during
607 /// operations that move a marker from one instruction to another.
608 Instruction *MarkedInstr = nullptr;
609
610 /// List of DbgRecords, the non-instruction equivalent of llvm.dbg.*
611 /// intrinsics. There is a one-to-one relationship between each debug
612 /// intrinsic in a block and each DbgRecord once the representation has been
613 /// converted, and the ordering is meaningful in the same way.
614 simple_ilist<DbgRecord> StoredDbgRecords;
615 bool empty() const { return StoredDbgRecords.empty(); }
616
617 LLVM_ABI const BasicBlock *getParent() const;
618 LLVM_ABI BasicBlock *getParent();

620 /// Handle the removal of a marker: the position of debug-info has gone away,
621 /// but the stored debug records should not. Drop them onto the next
